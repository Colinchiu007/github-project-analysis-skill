# GitHub 项目分析 Skill

> **中文名称**：GitHub 项目分析
> **用途**：针对某个需求和技术领域，在 GitHub 搜索相关项目进行分析，给出可行性复用报告
> **创建时间**：2026-06-29

---

## 一、目标

针对某个需求和技术领域，在 GitHub 搜索相关项目进行分析，给出可行性复用报告。

---

## 二、输入方式

### 4 种方式

| 方式 | 说明 | 示例 |
|------|------|------|
| **方式 1** | 提出产品需求、功能、技术实现方法、技术点，由 Skill 理解分析，给出多个搜索关键词清单 | "我想做一个多平台内容发布工具" |
| **方式 2** | 直接提供 1 个或多个关键词 | "multi-platform publish, social media scheduler" |
| **方式 3** | 提供包含关键词列表的文档 | 提供 .md 文件路径 |
| **方式 4** | 直接提供 1 个或多个 GitHub 项目 URL | "https://github.com/user/project" |

### 关键词生成规则

- 从需求中提取**核心功能**关键词
- 从技术实现中提取**技术栈**关键词
- 从产品类型中提取**产品分类**关键词
- 包含**中文关键词**（用于搜索中国平台相关项目）
- 生成**组合关键词**（用于精确搜索）

---

## 三、搜索条件或结果筛选条件

### Star 数

| 条件 | 说明 |
|------|------|
| **默认** | Star > 100 |
| **自定义范围** | 可指定范围，如 5000-10K、2000以下、5K以上 |

### 最近更新

- 最近更新 < 24 个月（默认）

---

## 四、具体流程

### 流程图

```
用户输入需求/关键词
        ↓
确定搜索关键词（3种方式）
        ↓
生成关键词清单并保存为 .md 文件
        ↓
用关键词依次在 GitHub 搜索
        ↓
按筛选条件筛选项目（Star 数、更新时间）
        ↓
保存所有项目 URL 列表
        ↓
对每个项目进行深入分析
        ↓
生成分析报告
```

### 详细步骤

#### 步骤 1：确定输入方式

**方式 1：从需求生成关键词**
- 分析用户需求
- 提取核心功能点
- 生成技术实现关键词
- 生成产品类型关键词
- 生成中文关键词
- 保存为 .md 文件

**方式 2：使用提供的关键词**
- 解析用户提供的关键词
- 验证关键词有效性
- 直接使用

**方式 3：从文档读取关键词**
- 读取 .md 文件
- 解析关键词列表
- 验证关键词有效性

**方式 4：直接提供 GitHub 项目 URL**
- 解析用户提供的 URL
- 验证 URL 有效性
- 直接进入项目分析（跳过搜索和筛选）

#### 步骤 2：GitHub 搜索（方式 1-3 执行）

- 使用 GitHub API 搜索
- 对每个关键词执行搜索
- 控制 API 速率限制（每分钟 10 次）
- 记录搜索结果

#### 步骤 3：筛选项目（方式 1-3 执行）

- 按 Star 数筛选（默认 > 100）
- 按最近更新时间筛选（< 24 个月）
- 去重（同一项目可能匹配多个关键词）

#### 步骤 4：保存项目列表（方式 1-3 执行）

- 保存所有符合条件的项目 URL
- 记录项目基本信息（名称、Star、语言、描述）
- 按 Star 数降序排列

#### 步骤 5：深入分析每个项目

对每个项目进行**详细、深入、多维度**的分析：

**产品需求分析（必须包含）**
- 项目名称和 URL
- Star 数、Fork 数、最近更新时间
- 核心功能（详细描述）
- 目标用户群体
- 产品定位和差异化
- 竞品对比（与目标项目的对比）
- 用户评价和社区活跃度

**技术架构分析（必须包含）**
- 技术栈详细说明（前端、后端、数据库、依赖库）
- 架构设计模式（单体/微服务/插件化等）
- 代码质量评估（代码规范、测试覆盖、文档质量）
- 扩展性评估（插件系统、API 设计、模块化程度）
- 性能表现（基准测试、内存占用、响应速度）
- 安全性评估（认证机制、数据加密、漏洞风险）

**技术实现分析（必须包含）**
- 关键技术点（核心算法、实现方案）
- 代码示例（关键代码片段）
- 依赖分析（第三方库、版本、许可证）
- 部署方式（Docker、二进制、源码编译）
- 维护状态（提交频率、Issue 响应、贡献者活跃度）

**功能点分析（必须包含）**
- 完整功能清单（按模块分类）
- 功能完整性评估（与目标项目的对比）
- 用户体验分析（界面设计、交互流程）
- 缺失功能分析（与目标项目的差距）

**市场机会分析（必须包含）**
- 市场需求分析（目标用户痛点）
- 竞争格局（直接竞品、间接竞品）
- 差异化机会（可以改进的方向）
- 商业模式（开源/付费/企业版）

**复用建议（必须包含）**
- 可直接复用的组件（代码、模块、配置）
- 可借鉴的设计（架构、算法、流程）
- 需要重新实现的部分（技术栈差异）
- 复用难度评估（高/中/低）
- 预计工作量（人天/人周）

#### 步骤 6：生成报告

**项目数 < 100 个时：**
- 生成一份完整的汇总报告
- 不删除、不精简
- 包含所有项目的详细分析

**项目数 >= 100 个时：**
- 每个项目单独生成报告
- 再进行综合总结
- 生成一份综合报告

## 五、输出成果

### 1. 关键词清单（方式 1-3）

```
GitHub-关键词清单-{日期}.md
```

内容：
- 关键词列表
- 关键词分类
- 搜索建议
- **每个关键词搜索出的项目数**
- **所有关键词搜索筛选后的项目总数**

### 2. 项目列表清单（方式 1-3）

```
GitHub-项目列表-{日期}.md
```

内容：
- 项目名称
- 项目 URL
- Star 数
- 语言
- 描述
- 最近更新时间
- **匹配的关键词**

**注意**：当使用方式 4（直接提供 GitHub 项目 URL）时，不输出此文档。

### 3. 分析报告

**项目数 < 100 个时：**
```
GitHub-项目分析报告-{日期}.md
```

**项目数 >= 100 个时：**
```
GitHub-项目分析报告-{日期}/
├── 项目1-分析报告.md
├── 项目2-分析报告.md
├── ...
├── 项目N-分析报告.md
└── 综合总结报告.md
```

## 八、注意事项

1. **API 限制**：GitHub API 有速率限制，搜索时需要控制频率
2. **去重**：同一项目可能匹配多个关键词，需要去重
3. **筛选条件**：Star 数和更新时间是默认筛选条件，可根据需要调整
4. **报告完整性**：项目数 < 100 时，报告必须完整，不能删除或精简
5. **分析深度**：每个项目都需要深入分析，不能只做表面描述
