# GitHub 项目分析 Skill

> **中文名称**：GitHub 项目分析
> **用途**：针对某个需求和技术领域，在 GitHub 搜索相关项目进行分析，给出可行性复用报告
> **创建时间**：2026-06-29

---

## 一、目标

针对某个需求和技术领域，在 GitHub 搜索相关项目进行分析，给出可行性复用报告。

---

## 二、输入方式

### 4 种方式

| 方式 | 说明 | 示例 |
|------|------|------|
| **方式 1** | 提出产品需求、功能、技术实现方法、技术点，由 Skill 理解分析，给出多个搜索关键词清单 | "我想做一个多平台内容发布工具" |
| **方式 2** | 直接提供 1 个或多个关键词 | "multi-platform publish, social media scheduler" |
| **方式 3** | 提供包含关键词列表的文档 | 提供 .md 文件路径 |
| **方式 4** | 直接提供 1 个或多个 GitHub 项目 URL | "https://github.com/user/project" |

### 关键词生成规则

- 从需求中提取**核心功能**关键词
- 从技术实现中提取**技术栈**关键词
- 从产品类型中提取**产品分类**关键词
- 包含**中文关键词**（用于搜索中国平台相关项目）
- 生成**组合关键词**（用于精确搜索）

---

## 三、搜索条件或结果筛选条件

### Star 数

| 条件 | 说明 |
|------|------|
| **默认** | Star > 100 |
| **自定义范围** | 可指定范围，如 5000-10K、2000以下、5K以上 |

### 最近更新

- 最近更新 < 24 个月（默认）

---

## 四、具体流程

### 流程图

```
用户输入需求/关键词
        ↓
确定搜索关键词（3种方式）
        ↓
生成关键词清单并保存为 .md 文件
        ↓
用关键词依次在 GitHub 搜索
        ↓
按筛选条件筛选项目（Star 数、更新时间）
        ↓
保存所有项目 URL 列表
        ↓
对每个项目进行深入分析
        ↓
生成分析报告
```

### 详细步骤

#### 步骤 1：确定输入方式

**方式 1：从需求生成关键词**
- 分析用户需求
- 提取核心功能点
- 生成技术实现关键词
- 生成产品类型关键词
- 生成中文关键词
- 保存为 .md 文件

**方式 2：使用提供的关键词**
- 解析用户提供的关键词
- 验证关键词有效性
- 直接使用

**方式 3：从文档读取关键词**
- 读取 .md 文件
- 解析关键词列表
- 验证关键词有效性

**方式 4：直接提供 GitHub 项目 URL**
- 解析用户提供的 URL
- 验证 URL 有效性
- 直接进入项目分析（跳过搜索和筛选）

#### 步骤 2：GitHub 搜索（方式 1-3 执行）

- 使用 GitHub API 搜索
- 对每个关键词执行搜索
- 控制 API 速率限制（每分钟 10 次）
- 记录搜索结果

#### 步骤 3：筛选项目（方式 1-3 执行）

- 按 Star 数筛选（默认 > 100）
- 按最近更新时间筛选（< 24 个月）
- 去重（同一项目可能匹配多个关键词）

#### 步骤 4：保存项目列表（方式 1-3 执行）

- 保存所有符合条件的项目 URL
- 记录项目基本信息（名称、Star、语言、描述）
- 按 Star 数降序排列

#### 步骤 5：深入分析每个项目

对每个项目进行分析：

**产品需求分析**
- 核心功能
- 目标用户
- 产品定位
- 竞品对比

**技术架构分析**
- 技术栈
- 架构设计
- 代码质量
- 扩展性

**技术实现分析**
- 关键技术点
- 实现方案
- 性能表现
- 安全性

**功能点分析**
- 功能清单
- 功能完整性
- 用户体验

**市场机会分析**
- 市场需求
- 竞争格局
- 差异化机会

#### 步骤 6：生成报告

**项目数 < 100 个时：**
- 生成一份完整的汇总报告
- 不删除、不精简
- 包含所有项目的详细分析

**项目数 >= 100 个时：**
- 每个项目单独生成报告
- 再进行综合总结
- 生成一份综合报告

- 搜索建议

```

```

内容：
- 项目名称
- 项目 URL
- Star 数
- 语言
- 描述
- 最近更新时间

**注意**：当使用方式 4（直接提供 GitHub 项目 URL）时，不输出此文档。

### 3. 分析报告

**项目数 < 100 个时：**
```
GitHub-项目分析报告-{日期}.md
```

**项目数 >= 100 个时：**
```
GitHub-项目分析报告-{日期}/
├── 项目1-分析报告.md
├── 项目2-分析报告.md
├── ...
├── 项目N-分析报告.md
└── 综合总结报告.md
```

## 七、使用示例

### 示例 1：从需求生成关键词

```
用户输入：我想做一个多平台内容发布工具，支持小红书、抖音、视频号、B站

Skill 输出：
1. 生成关键词清单（49个关键词）
2. 保存为 GitHub-关键词清单-2026-06-29.md
3. 依次搜索 GitHub
4. 筛选 Star > 100 的项目
5. 分析每个项目
6. 生成分析报告
```

### 示例 2：直接提供关键词

```
用户输入：搜索关键词：multi-platform publish, social media scheduler, video upload

Skill 输出：
1. 直接使用提供的关键词
2. 搜索 GitHub
3. 筛选项目
4. 分析项目
5. 生成报告
```

### 示例 3：从文档读取关键词

```
用户输入：使用 D:\Data\projects\GitHub搜索关键词清单.md 中的关键词

Skill 输出：
1. 读取文档
2. 解析关键词列表
3. 搜索 GitHub
4. 筛选项目
5. 分析项目
6. 生成报告
```

### 示例 4：直接提供 GitHub 项目 URL

```
用户输入：分析这些项目 https://github.com/user/project1, https://github.com/user/project2

Skill 输出：
1. 解析 URL
2. 直接分析每个项目
3. 生成分析报告（不输出关键词清单和项目列表）
```

---

## 八、注意事项

1. **API 限制**：GitHub API 有速率限制，搜索时需要控制频率
2. **去重**：同一项目可能匹配多个关键词，需要去重
3. **筛选条件**：Star 数和更新时间是默认筛选条件，可根据需要调整
4. **报告完整性**：项目数 < 100 时，报告必须完整，不能删除或精简
5. **分析深度**：每个项目都需要深入分析，不能只做表面描述
