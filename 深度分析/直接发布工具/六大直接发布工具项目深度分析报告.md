## 三、跨项目可复用技术点总结

### 3.1 社交平台集成架构（最高复用价值）

| 设计要素 | Postiz 方案 | Brightbean 方案 | 推荐复用 |
|---------|-----------|----------------|---------|
| Provider接口 | `SocialProvider` interface | `SocialProvider` ABC | **Brightbean** |
| 基类 | `SocialAbstract`（含fetch重试） | `SocialProvider._request()` | **合并两者** |
| 错误处理 | `RefreshToken`/`BadBody` 异常 | `RateLimitError`/`APIError` | **Brightbean** |
| 类型定义 | TypeScript interfaces | Python dataclasses | **Brightbean** |
| 并发控制 | `maxConcurrentJob` 属性 | `ThreadPoolExecutor` | **Brightbean** |

### 3.2 发布引擎设计

**推荐方案**：参照 Brightbean 的 `PublishEngine`

核心模式：
1. **轮询模式** — 每15秒扫描到期帖子
2. **分组并行** — 同一Post的多平台并行发布
3. **指数退避重试** — 60s → 300s → 1800s
4. **首条评论** — 延迟120秒发布
5. **Token自动刷新** — 发布前检查Token过期

### 3.3 错误处理与重试

| 项目 | 重试策略 | 退避时间 | 最大重试 | Token刷新 |
|------|---------|---------|---------|----------|
| Postiz | Temporal ActivityRetry | 5秒 | 2次 | `RefreshToken` 异常触发 |
| Brightbean | 指数退避 | 60s/300s/1800s | 3次 | 发布前自动刷新 |
| Mixpost | Laravel Queue Retry | 配置驱动 | 可配置 | `tokenIsAboutToExpire()` |

### 3.4 数据库设计对比

| 项目 | ORM | Post状态 | 平台账号模型 |
|------|-----|---------|------------|
| Postiz | Prisma (PostgreSQL) | `State` enum: QUEUE/DRAFT | `Integration` (token, refreshToken) |
| Mixpost | Eloquent (MySQL/PG) | `PostStatus` enum | `Account` (pivot with errors) |
| Brightbean | Django ORM | `PlatformPost.Status` | `SocialAccount` (EncryptedTextField) |

## 四、Multi-Publish 适用性评估

### 直接可用的代码

1. **Brightbean 的 Provider 基类和类型系统** — 18个平台的Provider实现可直接参考
2. **Postiz 的 Social Provider 接口** — 最完整的社交平台抽象
3. **Brightbean 的发布引擎** — 并行发布+重试+限流+Token刷新
4. **Mixpost 的 Post 状态模型** — 简洁的 `PostStatus` 枚举
5. **Postiz 的 SSRF 安全防护** — 可直接复用

### 需要适配的代码

1. **Postiz 的 Provider 实现** — 36个Provider需从TypeScript适配
2. **Social Media Agent 的 LangGraph 工作流** — 需重新设计状态节点
3. **YouTube Agent 的发布调度** — 排期逻辑可复用，API调用需适配

### 不建议复用的代码

1. **Youtarr** — 定位为下载管理，与发布需求差异大
2. **Postiz 的前端代码** — Next.js前端与后端发布逻辑无关
3. **Mixpost 的 Laravel 特定代码** — 需要Laravel框架
